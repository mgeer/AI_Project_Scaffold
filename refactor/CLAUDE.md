# CLAUDE.md — 遗留项目重构规范

适用：C++ 微服务架构，算法封装为独立 SO 库。

---

## 一、启动检查

检查 `project-knowledge/` 是否包含：
`structure.md` `symbol-index.md` `dependencies.md` `gotchas.md` `patterns.md` `progress.md`

缺失则提示用户初始化（见根目录 `INIT.md`），初始化前不开始任何任务。

---

## 二、Token 节约（强制）

**禁止：** 全项目扫描 / 重复读取已读文件 / 回复中大段复述代码（用 `file:line` 引用）

**定位流程（找到即停）：**
1. 查 `symbol-index.md`
2. 告知用户搜索关键词，由用户执行 grep
3. 只读相关函数/类，不读整个文件
4. 问题解决后更新 `gotchas.md`

**必须暂停提醒用户的情况：**
- 即将读取超过 3 个文件
- 任务范围超出当前模块
- 对话超过 15 轮
- 发现自己在猜测文件内容

**会话结束时提醒更新：** `progress.md` / `gotchas.md` / `symbol-index.md`

---

## 三、重构范围（严格模块隔离）

- 每次只处理一个微服务模块
- SO 库接口禁止修改
- 跨模块影响须拆成独立任务

**动代码前输出确认：**
```
目标模块 / 涉及文件 / SO 调用点 / gotchas 风险 / 对外接口 / 计划步骤
```
等用户确认后再执行。

---

## 四、测试策略

**三层测试：**
- 单元测试：Mock SO 库，测数据处理逻辑，TDD（先写测试再写实现）
- 契约测试：重构前建立，锁住微服务间接口格式
- 集成测试：真实 SO + 固定历史数据，容差断言

**TDD 顺序：** red → green → refactor → 提交。禁止先实现再补测试。

**覆盖要求：** 单元测试 ≥ 80%，所有 SO 调用点有 Mock 测试，异常路径有覆盖。

---

## 五、执行纪律

- 需要猜测时停下来提问
- 测试失败时分析根因，不绕过
- 确认无引用的死代码直接删除（删前 grep 全局确认，SO 导出函数需确认外部依赖）
- 每个 PR 只含一个模块变更
